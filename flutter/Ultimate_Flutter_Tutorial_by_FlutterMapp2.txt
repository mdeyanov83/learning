üó∫Ô∏è 7. Navigation
    Navigator() is a Flutter widget and class that manages a stack of routes (screens). It lets you navigate
    between pages, push new screens, and pop back to previous ones ‚Äî like a web browser‚Äôs history.
    * Stack-based navigation:
        - Each screen is a route.
        - You can push (go forward) or pop (go back).

    Navigator.push(
        context,
        MaterialPageRoute(builder: (context) => SecondPage()),
        );
    Navigator.pop(context);
    üß± Common methods:
        push()	            Opens a new route (screen) and adds it to the stack
        pop()	            Closes the current route and goes back
        pushReplacement()	Replaces the current route with a new one
        pushNamed() / popNamed()	Navigate using route names (defined in MaterialApp)
        canPop()	        Checks if there‚Äôs a route to go back to
    ‚úÖ Examples:
        1. Navigate to a new page:
            Navigator.push(
                context,
                MaterialPageRoute(builder: (context) => SecondScreen()),
            );
        2. Go back to the previous page:
            Navigator.pop(context);
        3. Navigate with named routes (recommended for larger apps):
            In MaterialApp:
            MaterialApp(
                initialRoute: '/',
                routes: {
                    '/': (context) => HomeScreen(),
                    '/second': (context) => SecondScreen(),
                },
            );
        Navigate:
            Navigator.pushNamed(context, '/second');
            Navigator.pop(context);
        4. Replace current screen:
            Navigator.pushReplacement(
                context,
                MaterialPageRoute(builder: (context) => LoginScreen()),
            );
    üìå Tip:
        Use Navigator.pushNamed() for cleaner navigation when you have multiple screens.
        For modern navigation, consider go_router or Navigator 2.0 (Router API) for complex apps.
        Navigator works perfectly with Scaffold for building multi-screen apps.

    02:02:34 Push
        Navigator.push() is used to navigate to a new screen (route) by pushing it onto the navigation stack.
        It shows the new screen on top of the current one, allowing the user to go back later with Navigator.pop().
        Navigator.push(
            context,
            MaterialPageRoute(builder: (context) => NextPage()),
        );
        üß± Parameters:
            context ‚Üí The current widget‚Äôs build context.
            route ‚Üí The route (page/screen) to display, usually a MaterialPageRoute.
        ‚úÖ Example 1 ‚Äì Navigate to another screen:
            // First screen
            ElevatedButton(
                onPressed: () {
                    Navigator.push(
                    context,
                    MaterialPageRoute(builder: (context) => SecondScreen()),
                    );
                },
                child: Text('Go to Second Screen'),
            );
            // Second screen
            class SecondScreen extends StatelessWidget {
                @override
                Widget build(BuildContext context) {
                    return Scaffold(
                        appBar: AppBar(title: Text('Second Screen')),
                        body: Center(
                            child: ElevatedButton(
                                onPressed: () {
                                    Navigator.pop(context); // Go back
                                },
                                child: Text('Back to First Screen'),
                            ),
                        ),
                    );
                }
            }
        ‚úÖ Example 2 ‚Äì Passing data to next screen:
            Navigator.push(
                context,
                MaterialPageRoute(
                    builder: (context) => DetailScreen(item: 'Flutter Rocks!'),
                ),
            );
            class DetailScreen extends StatelessWidget {
                final String item;
                DetailScreen({required this.item});
                @override
                Widget build(BuildContext context) {
                    return Scaffold(
                        appBar: AppBar(title: Text('Detail Screen')),
                        body: Center(child: Text(item)),
                    );
                }
            }
        ‚úÖ Example 3 ‚Äì Getting a result from the next screen:
            final result = await Navigator.push(
                context,
                MaterialPageRoute(builder: (context) => ChooseColorScreen()),
            );
            print('User selected: $result');
            Then, in ChooseColorScreen:
            Navigator.pop(context, 'Blue');
        üìå Tip:
            Use Navigator.pushNamed() if you‚Äôve defined named routes in MaterialApp.
            Always use Navigator.pop(context) to return to the previous screen.
            Navigator.push() returns a Future that completes when the new route is popped ‚Äî useful for receiving results back.

    02:02:34 Pop
        Navigator.pop() is used to close the current screen (or route) and go back to the previous one.
        It‚Äôs the reverse of Navigator.push().
        When called, it pops the top route off the navigation stack.

        Navigator.pop(context);

        üß± Parameters:
            context ‚Üí The current widget‚Äôs build context.
            result (optional) ‚Üí Data you can return to the previous screen.
        ‚úÖ Example 1 ‚Äì Simple Back Navigation:
            ElevatedButton(
                onPressed: () {
                    Navigator.pop(context);
                },
                child: Text('Go Back'),
            );
            ‚û°Ô∏è Returns to the previous screen that was pushed using Navigator.push().

        ‚úÖ Example 2 ‚Äì Returning data to previous screen:
            Second screen:
                ElevatedButton(
                    onPressed: () {
                        Navigator.pop(context, 'Hello from Second Screen!');
                    },
                    child: Text('Return Data'),
                );
            First screen:
                final result = await Navigator.push(
                    context,
                    MaterialPageRoute(builder: (context) => SecondScreen()),
                );
                print(result); // Output: Hello from Second Screen!
        ‚úÖ Example 3 ‚Äì Conditional pop:
            if (Navigator.canPop(context)) {
                Navigator.pop(context);
            }
            ‚û°Ô∏è Checks if there‚Äôs a previous screen before popping (avoids errors if at the root).
        üìå Tip:
            Navigator.pop() works only if a route was pushed before.
            You can return data by passing it as the second argument.
            Use it to close dialogs, modals, or bottom sheets too (not just pages).
            Example:
                showDialog(
                    context: context,
                    builder: (context) => AlertDialog(
                        title: Text('Confirm'),
                        actions: [
                            TextButton(
                                onPressed: () => Navigator.pop(context, false),
                                child: Text('Cancel'),
                            ),
                            TextButton(
                                onPressed: () => Navigator.pop(context, true),
                                child: Text('OK'),
                            ),
                        ],
                    ),
                );
                ‚û°Ô∏è The dialog returns true or false when popped!

    02:09:16 PushReplacement
        *** Navigator.pushReplacement() is used to navigate to a new screen while replacing the current one.
        That means the previous screen is removed from the navigation stack ‚Äî the user cannot go back using the back button.
        It‚Äôs often used for navigation flows like login ‚Üí home screen or splash ‚Üí main app.

        Navigator.pushReplacement(
            context,
            MaterialPageRoute(builder: (context) => NewScreen()),
        );
        üß± Parameters:
            context ‚Üí current widget‚Äôs BuildContext
            route ‚Üí the route (screen) to replace with
            result (optional) ‚Üí data returned to the replaced route
        ‚úÖ Example 1 ‚Äì Replace login with home screen:
            ElevatedButton(
                onPressed: () {
                    Navigator.pushReplacement(
                    context,
                    MaterialPageRoute(builder: (context) => HomeScreen()),
                    );
                },
                child: Text('Login'),
            );
            ‚û°Ô∏è The user goes to HomeScreen and can‚Äôt go back to LoginScreen.
        ‚úÖ Example 2 ‚Äì Replace with named route:
            Navigator.pushReplacementNamed(context, '/home');
        ‚úÖ Example 3 ‚Äì Replace and send back a result:
            Navigator.pushReplacement(
                context,
                MaterialPageRoute(builder: (context) => NextScreen()),
                result: 'Replaced!',
            );
        üìå Tip:
            Use pushReplacement() when you don‚Äôt want users to return to the previous screen.
            For example:
            After login ‚Üí home screen
            After splash ‚Üí main app
            After onboarding ‚Üí main content
            If you need to remove all previous routes, use:
            Navigator.pushAndRemoveUntil(
                context,
                MaterialPageRoute(builder: (context) => HomeScreen()),
                (route) => false, // removes all previous routes
            );

        *** Navigator.pushAndRemoveUntil() is used to navigate to a new screen and remove previous routes from the
        navigation stack until a given condition is met. This is useful for things like login flows ‚Äî where you
        want to go to a home screen and prevent the user from going back to the login page.

        Navigator.pushAndRemoveUntil(
            context,
            MaterialPageRoute(builder: (context) => NewScreen()),
            (Route<dynamic> route) => false,
        );

        üß± Parameters:
            context ‚Üí The build context.
            MaterialPageRoute ‚Üí The route (new page) to push.
            predicate ‚Üí A condition that decides which previous routes to keep.
            If you return false, all previous routes are removed.
            If you return true for a specific route, that route and the ones below it are kept.

        ‚úÖ Example 1 ‚Äì Remove all previous routes:
            Navigator.pushAndRemoveUntil(
                context,
                MaterialPageRoute(builder: (context) => HomeScreen()),
                (route) => false,
            );
        üìå This pushes HomeScreen and clears the entire stack. The user can‚Äôt go back.

        ‚úÖ Example 2 ‚Äì Keep the first route (like a splash or root):
            Navigator.pushAndRemoveUntil(
                context,
                MaterialPageRoute(builder: (context) => HomeScreen()),
                (route) => route.isFirst,
            );
        üìå This keeps only the first screen in the stack and removes everything else.

        ‚úÖ Example 3 ‚Äì Conditional removal:
            Navigator.pushAndRemoveUntil(
                context,
                MaterialPageRoute(builder: (context) => Dashboard()),
                (route) => route.settings.name == '/login',
            );
        üìå This keeps only the /login route and removes others.

        üß† When to use:
            After successful login/signup (to prevent going back to auth screen).
            To reset the navigation stack after logout or onboarding.
            To redirect users to a main page and clear history.
        üìå Tip
            Similar to pushReplacement(), but more powerful because it can remove multiple previous routes.
            Make sure to set up RouteSettings names if you plan to use named predicates.

    02:13:18 ClipRRect
        ClipRRect is a widget that clips its child using a rounded rectangle.
        It‚Äôs commonly used to give widgets like images, containers, or cards
        rounded corners. RRect stands for ‚Äúrounded rectangle‚Äù.

        ClipRRect(
            borderRadius: BorderRadius.circular(12),
            child: Image.asset('assets/image.png'),
        );

        üß± Common properties:
            borderRadius ‚Üí controls the roundness of the corners
            clipBehavior ‚Üí defines how clipping is applied (e.g., Clip.antiAlias, Clip.hardEdge)
            child ‚Üí the widget to be clipped
        ‚úÖ Examples:
            1. Rounded image:
                ClipRRect(
                    borderRadius: BorderRadius.circular(20),
                    child: Image.asset(
                        'assets/photo.jpg',
                        width: 200,
                        height: 200,
                        fit: BoxFit.cover,
                    ),
                );
            2. Rounded container:
                ClipRRect(
                    borderRadius: BorderRadius.circular(30),
                    child: Container(
                        color: Colors.blue,
                        width: 150,
                        height: 150,
                    ),
                );
            3. With custom radius per corner:
                ClipRRect(
                    borderRadius: BorderRadius.only(
                        topLeft: Radius.circular(20),
                        bottomRight: Radius.circular(40),
                    ),
                    child: Container(
                        color: Colors.green,
                        width: 150,
                        height: 150,
                    ),
                );
        üìå Tip:
            ClipRRect only clips the visible area ‚Äî it doesn‚Äôt affect the layout.
            For rounded corners on Container backgrounds, you can also use BoxDecoration with borderRadius.
            Use ClipOval if you want to make a circle or ellipse shape instead.

    02:17:40 Hero()
        Hero() is a widget that provides a shared element transition between two routes (screens).
        When you navigate to another page, the Hero widget animates the transition of a matching
        widget smoothly between the screens.
        This is great for image previews, profile pictures, or cards that expand into detail pages.

        Hero(
            tag: 'hero-tag',
            child: Image.asset('assets/image.png'),
        );

        ‚ö° Important:
            Both the source and destination Hero must have the same tag.
            The tag must be unique for each Hero pair.
        üß± Common properties:
            tag ‚Üí unique identifier to match Hero widgets between screens.
            child ‚Üí the widget to animate.
            flightShuttleBuilder ‚Üí optional, allows customizing the animation look.
            transitionOnUserGestures ‚Üí allows transitions during back gesture (iOS).
        ‚úÖ Example 1 ‚Äì Simple Hero animation:
            First screen:
                GestureDetector(
                    onTap: () {
                        Navigator.push(
                        context,
                        MaterialPageRoute(builder: (context) => SecondScreen()),
                        );
                    },
                    child: Hero(
                        tag: 'profile',
                        child: Image.asset(
                        'assets/avatar.png',
                        width: 100,
                        ),
                    ),
                );
            Second screen:
                Hero(
                    tag: 'profile',
                    child: Image.asset(
                        'assets/avatar.png',
                        width: 300,
                    ),
                );
            ü™Ñ When you tap the image, it smoothly grows from 100px to 300px as the new screen opens.
        ‚úÖ Example 2 ‚Äì Custom transition using flightShuttleBuilder:
            Hero(
                tag: 'logo',
                flightShuttleBuilder: (flightContext, animation, direction, fromContext, toContext) {
                    return RotationTransition(
                    turns: animation,
                    child: Image.asset('assets/logo.png'),
                    );
                },
                child: Image.asset('assets/logo.png'),
            );
        üìå Tip:
            Hero animations work best with images or simple widgets.
            Always use the same tag on both widgets.
            Works automatically with Navigator.push() and Navigator.pop().
            You can wrap any widget in a Hero‚Äînot just images!
        üß≠ Common use cases:
            Profile avatar ‚Üí profile page
            Thumbnail image ‚Üí fullscreen view
            Card preview ‚Üí detail screen

    02:22:14 SnackBar
        SnackBar is a lightweight message displayed at the bottom of the screen to provide
        brief feedback to the user. It can include a simple message, an action (like ‚ÄúUNDO‚Äù),
        and disappears automatically after a few seconds.

        ScaffoldMessenger.of(context).showSnackBar(
            SnackBar(
                content: Text('This is a SnackBar!'),
            ),
        );

        üß± Common properties:
            content ‚Üí the main widget, usually a Text.
            backgroundColor ‚Üí sets the background color of the snackbar.
            action ‚Üí adds an optional button (e.g., ‚ÄúUndo‚Äù).
            duration ‚Üí how long it stays visible.
            behavior ‚Üí e.g., SnackBarBehavior.floating to make it float above UI.
            shape ‚Üí gives rounded corners or custom shapes.
        ‚úÖ Example 1 ‚Äì Basic SnackBar:
            ScaffoldMessenger.of(context).showSnackBar(
                SnackBar(
                    content: Text('Item added to cart'),
                ),
            );
        ‚úÖ Example 2 ‚Äì With action button:
            ScaffoldMessenger.of(context).showSnackBar(
                SnackBar(
                    content: Text('Message deleted'),
                    action: SnackBarAction(
                    label: 'UNDO',
                    onPressed: () {
                        // Undo action here
                    },
                    ),
                ),
            );
        ‚úÖ Example 3 ‚Äì Floating & styled SnackBar:
            ScaffoldMessenger.of(context).showSnackBar(
                SnackBar(
                    content: Text('Saved successfully!'),
                    backgroundColor: Colors.green,
                    behavior: SnackBarBehavior.floating,
                    shape: RoundedRectangleBorder(
                    borderRadius: BorderRadius.circular(10),
                    ),
                    duration: Duration(seconds: 3),
                ),
            );
        üìå Tip:
            Always use ScaffoldMessenger.of(context) instead of the old Scaffold.of(context) (deprecated).
            SnackBar disappears automatically, but you can also dismiss it programmatically.
            Great for quick feedback like ‚ÄúSaved‚Äù, ‚ÄúCopied‚Äù, ‚ÄúDeleted‚Äù, etc.

    02:25:29 Alert - showDialog(), AlertDialog(), AboutDialog(), showAboutDialog()

        *** showDialog() is a built-in function used to display a modal dialog on top of the current screen.
        It blocks interaction with the rest of the app until the dialog is dismissed.
        It‚Äôs commonly used with widgets like AlertDialog or SimpleDialog.

        showDialog(

            builder: (BuildContext context) {

                    title: Text('Dialog Title'),

                            child: Text('Close'),

                );
            },
        );

            builder ‚Üí returns the widget to display (e.g., AlertDialog).
            barrierDismissible ‚Üí whether tapping outside closes the dialog (true by default).
            barrierColor ‚Üí color behind the dialog (e.g., semi-transparent black).
            useSafeArea ‚Üí ensures it doesn‚Äôt overlap system UI.
            useRootNavigator ‚Üí decides which navigator to use.
        ‚úÖ Example 1 ‚Äì Simple AlertDialog:
            showDialog(
                context: context,
                builder: (context) {
                    return AlertDialog(
                        title: Text('Hello'),
                        content: Text('This is a simple dialog.'),
                        actions: [
                            TextButton(
                            onPressed: () => Navigator.pop(context),
                            child: Text('OK'),
                            ),
                        ],
                    );
                },
            );
        ‚úÖ Example 2 ‚Äì Non-dismissible dialog:
            showDialog(
                context: context,
                barrierDismissible: false,
                builder: (context) {
                    return AlertDialog(
                        title: Text('Wait'),
                        content: Text('You must choose an option.'),
                        actions: [
                            TextButton(
                            onPressed: () => Navigator.pop(context),
                            child: Text('OK'),
                            ),
                        ],
                    );
                },
            );
        ‚úÖ Example 3 ‚Äì Custom widget inside dialog:
            showDialog(
                context: context,
                builder: (context) {
                    return Dialog(
                        shape: RoundedRectangleBorder(
                            borderRadius: BorderRadius.circular(20),

                            padding: const EdgeInsets.all(16),

                                mainAxisSize: MainAxisSize.min,

                                    Text('Custom Dialog', style: TextStyle(fontSize: 18)),

            You can also use showModalBottomSheet() for bottom popups.
        üß≠ Common use cases:
            Confirmation dialogs (e.g., Delete / Cancel)
            Error or success messages
            Custom forms or widgets inside a popup

        *** InitState()
        initState() is a lifecycle method of the State class in Flutter.
        It is called exactly once when a StatefulWidget is inserted into
        the widget tree, before the widget is displayed.
        You typically use it to:
            Initialize variables
            Set up listeners or controllers
            Fetch initial data

        @override
        void initState() {
            super.initState();
            // Initialization code here
        }

        üß± Key points:
            Always call super.initState() first inside the method.
            Runs before the build() method.
            Called only once during the lifecycle of the state object.
            Good place for setting up things like TextEditingController, AnimationController, or API calls.
        üìå Tip:
            Do not use BuildContext to access InheritedWidget (like Theme.of) directly inside initState() ‚Äî use WidgetsBinding.instance.addPostFrameCallback if needed after build.
            Use initState() for one-time setup logic only ‚Äî not for UI updates.
        üß≠ Lifecycle Order (Simplified):
            initState() ‚Üí setup once ‚úÖ
            build() ‚Üí builds UI üß±
            didUpdateWidget() ‚Üí when widget changes üîÅ
            dispose() ‚Üí cleanup üßº

        ‚úÖ Example 1 ‚Äì Initializing a variable:

            class MyWidgetState extends State<MyWidget> {
                String message = '';
                @override
                void initState() {
                    super.initState();
                    message = 'Hello from initState!';
                }
                @override
                Widget build(BuildContext context) {
                    return Text(message);
                }
            }

        ‚úÖ Example 2 ‚Äì Setting up a controller:

            class MyWidgetState extends State<MyWidget> {
                final TextEditingController _controller = TextEditingController();
                @override
                void initState() {
                    super.initState();
                    _controller.text = 'Initial text';
                }
                @override
                void dispose() {
                    _controller.dispose(); // Always dispose controllers!
                    super.dispose();
                }
                @override
                Widget build(BuildContext context) {
                    return TextField(controller: _controller);
                }
            }

        ‚úÖ Example 3 ‚Äì Fetching data before UI builds:

            @override
            void initState() {
                super.initState();
                fetchData();
            }
            void fetchData() async {
                // Simulate API call
                await Future.delayed(Duration(seconds: 2));
                print('Data loaded');
            }

        *** dispose()
        dispose() is a lifecycle method of the State class that is called when a StatefulWidget
        is permanently removed from the widget tree. It‚Äôs used to clean up resources such as
        controllers, listeners, animations, or streams to prevent memory leaks.

        @override
        void dispose() {
            // Clean up resources here
            super.dispose();
        }

        üß± Key points:
            Called only once when the state object is removed.
            Always call super.dispose() at the end of the method.
            Ideal for disposing controllers like TextEditingController, AnimationController, FocusNode, or streams.
        üìå Tip:
            Always clean up what you initialize in initState().
            Forgetting to dispose can cause memory leaks and performance issues.
            dispose() is never called if the app is force-closed, but always when a widget is removed properly from the tree.
        üß≠ Lifecycle Order Recap:
            initState() ‚Üí Initialize resources.
            build() ‚Üí Render UI.
            didUpdateWidget() ‚Üí Respond to updates.
            dispose() ‚Üí Clean up resources. ‚úÖ

        ‚úÖ Example 1 ‚Äì Disposing a controller:

            class MyWidgetState extends State<MyWidget> {
                final TextEditingController _controller = TextEditingController();
                @override
                void dispose() {
                    _controller.dispose(); // important to prevent memory leaks
                    super.dispose();
                }
                @override
                Widget build(BuildContext context) {
                    return TextField(controller: _controller);
                }
            }

        ‚úÖ Example 2 ‚Äì Disposing an animation:

            class MyAnimationWidget extends State<MyWidget> with SingleTickerProviderStateMixin {
                late AnimationController _animationController;
                @override
                void initState() {
                    super.initState();
                    _animationController = AnimationController(
                    vsync: this,
                    duration: Duration(seconds: 2),
                    )..repeat();
                }
                @override
                void dispose() {
                    _animationController.dispose(); // stop and clean up animation
                    super.dispose();
                }
                @override
                Widget build(BuildContext context) {
                    return FadeTransition(
                    opacity: _animationController,
                    child: Text('Animating...'),
                    );
                }
            }

        ‚úÖ Example 3 ‚Äì Canceling a StreamSubscription:

            late StreamSubscription<int> _subscription;
            @override
                void initState() {
                super.initState();
                _subscription = Stream.periodic(Duration(seconds: 1), (i) => i)
                    .listen((event) => print(event));
            }
            @override
                void dispose() {
                _subscription.cancel();
                super.dispose();
            }

        *** OTHER COMMON OVERRIDES

        üß± 1Ô∏è‚É£ build(BuildContext context) ‚Äî (always overridden)

            @override
            Widget build(BuildContext context) {
                return Container();
            }

            Required for State classes.
            Called every time the widget needs to be rebuilt.
            Should be pure ‚Äî no side effects (like network calls).

        üß≠ 2Ô∏è‚É£ didChangeDependencies()

            @override
            void didChangeDependencies() {
                super.didChangeDependencies();
                // Called after initState, and whenever an InheritedWidget the state depends on changes
            }
            üß† Use case:
                Responding to changes in inherited widgets (like MediaQuery, Theme, Localizations).
                Good place for code that relies on context (which isn‚Äôt fully available in initState() yet).
            üëâ Common example:
                @override
                void didChangeDependencies() {
                    super.didChangeDependencies();
                    final mediaQuery = MediaQuery.of(context);
                }

        üîÅ 3Ô∏è‚É£ didUpdateWidget(covariant MyWidget oldWidget)
            @override
            void didUpdateWidget(MyWidget oldWidget) {
                super.didUpdateWidget(oldWidget);
                if (oldWidget.someValue != widget.someValue) {
                    // React to updated widget configuration
                }
            }
            üß† Use case:
                When parent rebuilds the widget with new parameters (but the same State object).
                Ideal for reacting to updated props without recreating the whole state.
            üëâ For example:
                if (oldWidget.userId != widget.userId) {
                    _loadUser(widget.userId);
                }

        üß™ 4Ô∏è‚É£ deactivate()
            @override
            void deactivate() {
                super.deactivate();
                // Called when the State is removed from the tree (but may be reinserted later)
            }
            üß† Use case:
            Cleanup when the widget is temporarily removed (e.g., during a navigation or reparenting).
            Rarely used compared to dispose(), but useful in some complex layouts.

        üßπ 5Ô∏è‚É£ reassemble() (debug only)
            @override
            void reassemble() {
            super.reassemble();
            // Called on hot reload in debug mode
            }
            üß† Use case:
                Used during hot reload only.
                Handy for debugging, reloading assets, resetting timers, etc.
            üëâ Example:
                @override
                void reassemble() {
                    super.reassemble();
                    print('Hot reload triggered');
                }

        üß† 6Ô∏è‚É£ (Less common but valid) setState()
            You don‚Äôt usually override it, but you can if you want to intercept state changes (e.g., logging or batching).
            @override
            void setState(VoidCallback fn) {
                super.setState(fn);
                print('State changed!');
            }
            ‚ö†Ô∏è Use carefully ‚Äî this affects all setState calls in that widget.

        7Ô∏è‚É£ Full State lifecycle overview
            Order of execution:
                createState() (in the widget)
                initState() ‚úÖ
                didChangeDependencies() ‚úÖ
                build() ‚úÖ (can be called many times)
                didUpdateWidget() (if parent updates props) ‚úÖ
                deactivate() (if removed from tree)
                dispose() ‚úÖ

            Also:
                reassemble() on hot reload
                setState() whenever state updates

        8Ô∏è‚É£ Quick Summary Table
            Method	                When it‚Äôs called	                                Common use cases
            ------                  ----------------                                    ----------------
            initState()	            Once when State is created	                        Init controllers, listeners, load data
            didChangeDependencies() After initState + when inherited widgets change     Access context-dependent data
            build()	                On every rebuild	                                UI building (no side effects)
            didUpdateWidget()	    When parent passes new widget with updated fields   Respond to prop changes
            deactivate()	        When removed from tree                              Advanced cleanup, reparenting
            dispose()	            When State is permanently removed	                Dispose controllers, cancel streams
            reassemble()	        On hot reload (debug only)	                        Debugging, reloading assets
            setState() (override)   On state change                                     Debugging, logging

        ‚úÖ In practice, the most commonly overridden methods besides are
            build() (always)
            initState()
            dispose()
            didChangeDependencies() ‚Üí for context-related setup
            didUpdateWidget() ‚Üí for reacting to prop changes
